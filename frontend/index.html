Verify that a release's changelog
matches the code it shipped.

VeritaLog fetches a repository's published changelog and the actual git diff between two tags, has independent GenLayer validators judge whether the changelog describes the diff, and records the verdict on-chain. No single party is trusted in the pipeline.

/ @

The gap

A changelog is written by the same party that controls the code. There is no separation of duties, so a release can understate or omit what actually changed. Downstream consumers read the changelog to decide whether to upgrade, with no independent check.

The XZ Utils 5.6.0 release contained a backdoor. Its release notes said nothing meaningful about it. Signing tools like cosign prove who published an artifact. They do not evaluate whether the release notes are honest about the diff. That semantic gap is what VeritaLog closes.

How it works

  1. 01

    Submit coordinates

    A requester provides the repository, the before tag, the after tag, and the changelog URL. VeritaLog derives the diff URL from GitHub's public compare endpoint.

  2. 02

    Fetch evidence

    The leader validator fetches the unified diff and the changelog text directly from the web. Both are public, so every validator can fetch the same sources independently.

  3. 03

    Judge the match

    An LLM reads the diff and changelog as evidence and returns one verdict: ACCURATE, PARTIAL, or MISLEADING, with a short reason.

  4. 04

    Reach consensus

    Validators re-run the judgment and agree only when the verdict matches. The reason text may differ. The agreed verdict is stored on-chain and can be appealed.

The verdict scale

Consensus is reached on this enum, not on prose. These four values are the entire vocabulary of an attestation.

ACCURATE

The changelog adequately describes the code changes.

PARTIAL

The changelog mentions some changes but omits significant ones.

MISLEADING

The changelog misrepresents or hides critical changes.

UNVERIFIED

No attestation exists yet for this repository and version.

Verify an attestation

Read the on-chain verdict for any repository and version. This is a read call. It needs no wallet and costs nothing.

Request an attestation

Submit a release for judgment. This is a write. It requires a browser wallet on Testnet Bradbury and sends one transaction. The verdict is decided by validators, not by this page.

Integrate

Gate a dependency upgrade in CI on the on-chain verdict using the GenLayer JS SDK. Reads require no key. 

import { createClient } from "genlayer-js";
import { testnetBradbury } from "genlayer-js/chains";

const client = createClient({ chain: testnetBradbury });

const a = await client.readContract({
  address: CONTRACT_ADDRESS,
  functionName: "get_attestation",
  args: ["acme", "widget", "v2.0.0"],
});

if (a.verdict === "MISLEADING" || !a.exists) {
  process.exit(1); // block the upgrade
}
